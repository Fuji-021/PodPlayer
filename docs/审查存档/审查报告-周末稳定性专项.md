## 一、参考材料的提取（相关 vs 不适用）

那份材料是按 **UWP / .NET / WinRT** 写的，与本项目（**Electron 13 + Chromium 91 + howler.js**）技术栈不符，约一半是"扯淡"——需剔除：

- **不适用（API/模型不对口）**：`SMTC`/`RandomAccessStreamReference`/`PlaybackInfo`、`backgroundAudio` 清单 + `Suspending`、`TimeTrigger`/`MaintenanceTrigger`、`WM_DPICHANGED`、`UISettings.ColorValuesChanged`、`AudioGraph`/`Media Foundation`、`.NET` 异常类、UWP 后台进程生命周期（Electron 是常规桌面进程，不受 UWP 沙箱管制）。

- **映射后相关（本轮据此审）**：系统媒体集成→`navigator.mediaSession`（媒体键/系统卡片）；生命周期→`powerMonitor`（睡眠）、退出持久化、托盘、窗口状态、主题；稳定性测试点→刷新风暴并发/DB、长期内存泄漏、坏文件/坏 feed 容错、磁盘满/权限/DB 损坏、睡眠恢复（即你说的"程序初始化/性能/内存压力/鲁棒性/系统耦合性"）。

## 二、新发现（按严重度）

### 🔴 P1

**P1-1 ｜ 下载写入流无 `error` 监听：磁盘满 / 目录不可写会崩主进程**（★最该先修）
- 位置：`src/electron/podcastDownload.js:146`（`createWriteStream`）。只挂了 `req.on('error')`、`res.on('error')`，**唯独没有 `writeStream.on('error')`**；主进程也无 `uncaughtException` 兜底。
- 触发/影响：下载中磁盘写满（ENOSPC）或目标目录被删/只读（EACCES/ENOENT）→ stream 抛无监听器的 `'error'` → Node 直接 throw → **整个 Electron 主进程崩溃**；即便不崩也不发 `download:error`，该集**永久卡 downloading** 并留截断/0 字节孤儿。
- 为何新：buglist 的"重启下载孤儿"是清库后的历史孤儿；这是**运行期写入失败**这条全新路径。

**P1-2 ｜ 每秒 `saveEpisodeProgress` 无 `catch`：IndexedDB 写失败每秒一条未捕获 rejection**（★最该先修）
- 位置：`src/utils/Player.js:339`（裸 `db.episodeProgress.put` promise 无 `.catch`），对照紧邻的 `tickListen().catch(()=>{})`（:389）有兜底。无全局 `unhandledrejection` 处理器。
- 触发/影响：IndexedDB 进入异常态（库锁争用/磁盘损坏/配额耗尽/versionchange）时**每秒**一条未捕获 rejection，刷爆控制台、某些配置下被当致命错误。落在**之前 IndexedDB 事故的同一故障域**——`openDatabase` 只加固了打开期，这条高频写路径没等同防护。
- 为何新：buglist"心跳每秒2写"讲写**次数**；这条讲写**失败无捕获**。

**P1-3 ｜ 无 `powerMonitor`：睡眠唤醒后播放/下载/设备均无恢复逻辑**
- 位置：缺失（`src/` 全工程 `grep powerMonitor/suspend/resume` 零命中）。
- 触发/影响：① 睡眠唤醒后本地 `file://`/普通 CDN 源 howler 常卡死（NAS 源有 3 秒看门狗兜底，但 `_nasSourceActive=false` 的源不覆盖）；② 下载流被掐后唤醒可能**永久卡"下载中"**，无 resume 重试；③ 无 `navigator.mediaDevices.devicechange` 监听，播放途中拔掉输出设备（蓝牙耳机断连）不自动回落默认设备 → 声音消失。
- 为何新：buglist 无任何睡眠/设备热插拔条目。（注：需真机实测确认 Chromium 91 在本项目加载方式下的实际行为）

**P1-4 ｜ 退出 / `app.exit()` / 崩溃前无 flush 钩子，Dexie 写可能半途夭折**
- 位置：`src/background.js:536-547`（`before-quit/will-quit/quit` 只做 `willQuitApp`/`unregisterAll`/`express.close`，不通知渲染端 flush）；全工程无 `beforeunload`/`pagehide`。持久化全靠每秒定时器。
- 触发/影响：退出按钮/托盘退出走 `app.exit()` **立即终止、不触发渲染进程 unload**，正在 `await` 的 Dexie 写（进度/统计）可能半途夭折——又是"IndexedDB 写中断"类隐患；正常崩溃/断电丢最后未满 1 秒进度。
- 为何新：与"心跳每秒2写"方向相反，是**退出路径缺最终一致性保障**。

**P1-5 ｜ Windows 物理媒体键（窗口失焦/托盘态）+ 系统"正在播放"卡片大概率失效**
- 位置：`src/utils/shortcuts.js` / `src/electron/globalShortcut.js`（只注册 `Alt+Ctrl+方向键` 等自定义组合，**未注册物理媒体键** `MediaPlayPause`/`MediaTrackNext`/`MediaTrackPrevious`）；物理媒体键唯一兜底是 `Player.vue:1803` 页内 `keydown`，**仅窗口聚焦时有效**；Chromium MediaSessionService 仅 Linux 关闭、Windows 放任默认。
- 触发/影响：最小化到托盘后用键盘媒体键控制播放（播客最常见用法）大概率无反应；锁屏/任务栏缩略图无"正在播放"卡片与封面。
- 为何新：buglist"mpris 监听不移除"是 Linux MPRIS 泄漏，这是 **Windows 平台媒体键/SMTC 能力缺口**。（需真机实测确认 SMTC 卡片是否出现）

**P1-6 ｜ 主进程 RSS 抓取无大小上限 + IPC 克隆双份，大档/海量刷新内存尖峰**
- 位置：`src/electron/podcastFetch.js:10-21`（`fetchText` 无 `maxContentLength`/`maxBodyLength`、无自定义 agent），消费侧 `service.js:171-175`。
- 触发/影响：大档 RSS 可达数十 MB（多年全量单集），`responseType:'text'` 无上限读进主进程，再经 `ipcRenderer.invoke` 完整结构化克隆复制到渲染端（瞬时双份）；高峰 5 档并行 = 5×2 份大字符串同驻 → 内存尖峰。
- 为何新：buglist 无 RSS 抓取大小/IPC 克隆条目。

**P1-7 ｜ `parseRss` 在渲染端主线程同步执行，批量刷新/导入阻塞 UI**
- 位置：`src/utils/podcast/service.js:176`（refresh）/`:41`（subscribe）/`:131`（importRss），实现 `rssParser.js` 同步。
- 触发/影响：一档几千单集的 feed 解析达数百 ms，期间独占主线程；2000 档刷新 / 大 OPML 导入时持续掉帧、点击无响应，与沉浸页大 blur 背景叠加更明显。建议移 Web Worker 或分片 `requestIdleCallback`。
- 为何新：buglist 的虚拟化/动画是渲染域；这是**解析管线阻塞主线程**。

**P1-8 ｜ 沉浸页背景叠两层重型滤镜 + 噪点混合，长开/低端机持续合成开销**（新功能引入）
- 位置：`src/components/Player.vue:2650-2678`（`.imm-bg-cover` `blur(64px) saturate(1.15)` 放大封面 **再叠** `.imm-bg-frost` 全屏 `backdrop-filter: blur(24px)` + `::after` `mix-blend-mode: overlay` 噪点）。
- 触发/影响：背景静态（取色已缓存+竞态保护，这部分做得好），但 `backdrop-filter` **每帧对背后内容重采样模糊**；沉浸页内任何动画（封面 play/pause scale、进度条、nyancat、标记充能圈）触发重绘时 frost 层反复重算 → 低端集显/4K 屏发烫、过渡掉帧。建议去掉 backdrop-filter 二次模糊（底层已 blur64）或降半径。
- 为何新：沉浸页是本次新功能，buglist 无其性能条目。（需低端机实测）

### 🟠 P2

- **P2-1 mediaSession 从不设 `playbackState`**：暂停后系统卡片仍显示"播放中"图标，部分 Windows 版无 `playbackState` 不展示卡片。缺失（`grep playbackState` 零命中）。
- **P2-2 mediaSession 重复注册 + play/pause 语义反转**：`Player.js:860` 与 `Player.vue:1786` 各注册一遍 handler，生效的 Player.vue 版把 `play`/`pause` 都绑 `playOrPause`(toggle) → 系统已暂停时发 `pause` 反被 toggle 成播放。
- **P2-3 运行时切系统深浅色，应用主体+托盘图标不实时跟随**：`common.js:115` 靠渲染端 `matchMedia` change；主进程无 `nativeTheme.on('updated')`，"自动"模式下切 Windows 主题常要重启才更新。
- **P2-4 最大化状态不恢复 + 最大化 bounds 污染还原尺寸**：`background.js:357-403` 只存/读 `getBounds()`，不存 `isMaximized`；且最大化时 `resized/moved` 把最大化 bounds 写进还原尺寸。
- **P2-5 `saveLyricFinished` 监听每次切歌重复注册不移除**（渲染端）：`Player.js:945` 每次切歌 `ipcRenderer.on` 不 removeListener，仅 OSDLyrics 开+网易云曲触发，累积后 IPC 放大 + MaxListeners 警告。（与 buglist 的主进程 mpris 监听不同位置）
- **P2-6 `upsertEpisodes` 读写无内部容错**：`db.js:151-167` 靠各调用方 try/catch 兜底，防御纵深不足，新增不包 catch 的调用方即回归白屏。
- **P2-7 RSS 解析无 item 数量 / description 长度上限**：`rssParser.js:82-117` 超大 feed（上万集、单条内嵌数 MB base64 图）全量入 IndexedDB + `v-html` 渲染巨型 DOM，逼近配额、拖慢全表扫描。
- **P2-8 OPML 宽容正则对超大输入 O(n·m) 卡顿**：`rssParser.js:154-170` 全文 `exec` 循环 + `[^>]*` 回溯，数十 MB 畸形 OPML 卡 UI 数秒（非指数 ReDoS）。
- **P2-9 OPML 导入纯串行（非 runLimited）+ source 漏标**：`service.js:92-103` `for await` 串行 + 单档 20s 超时拖死队列，2000 档实际不可用；且未传 `source` 全记 `'manual'`。（反直觉：不是太猛，是太慢）
- **P2-10 `tickListen` 每秒 fire-and-forget 读改写非原子，竞态丢统计**：`listening.js:34-101` `get→改→put` 非事务，且 `listenDaily` 第二段独立 R-M-W，seek/倍速快触发时后写覆盖前写、少计。（与"心跳每秒2写"是不同维度：这是丢更新）
- **P2-11 NAS `ensureItems/ensureEps` 无在途去重 + `limit=500` 硬上限**：`nasBridge.js:119-153` 冷缓存时 resolve/warm/podcastSet 三个入口对同一 `items?limit=500` 发 3 份重复请求；库超 500 档静默漏档（匹配不到即静默落 CDN）。

### 🟡 P3

- **P3-1 关闭策略三处重复 + Mac `exitAsk` 用 `minimize()` 而非 `hide()`**：`background.js` / `ipcMain.js:201-218` 逻辑分散，Mac 托盘语义不一致（Windows 路径正确）。
- **P3-2 多显示器越界判定逻辑 bug**：`background.js:280-295` 写成 `y < bounds.y - bounds.height`（恒 false），副屏窗口重启常被拉回主屏。
- **P3-3 封面取色缓存无上限（两套）**：`coverColor.js:6,47` + `coverPalette.js:14,110` 模块级 Map 只增不淘汰、无 LRU；连播跨数百集累积（单条小，实际几乎不到 MB，但 100% 单调）。建议套用同目录 `episodeCache.js` 已有的 LRU(cap 12)。
- **P3-4 NAS 心跳 `setInterval` 仅停用时清**：`nasSource.js:53-57` 无卸载清理，渲染端重载边缘场景可能叠加游离 interval（危害极小）。
- **P3-5 缺 `onplayerror` 兜底，加载成功后播放期失败静默卡住**：`Player.js:510-540` 只配 `onloaderror`；"加载 OK 但 body 损坏/中途断流/decode 失败"走 `playerror` 无人接 → 不报错不跳集（loaderror 路径已覆盖得很好，仅此窄缝）。

## 三、修复优先级 Top（性价比排序）

| 序 | 项 | 成本 | 收益 |
|---|---|---|---|
| 1 | **P1-1** writeStream 补 `.on('error')`（+ 主进程 `uncaughtException` 兜底） | 极低（几行） | 防主进程崩溃 + 清孤儿 + 解卡 |
| 2 | **P1-2** `saveEpisodeProgress` 补 `.catch` + 全局 `unhandledrejection` | 极低 | 堵住事故故障域的每秒 rejection |
| 3 | **P1-4** `before-quit` 通知渲染端 flush + `beforeunload` 兜底 | 中 | 退出/崩溃不丢进度（呼应数据丢失事故） |
| 4 | **P1-8** 沉浸页去掉 `backdrop-filter` 二次模糊或降半径 | 低 | 低端机/长开性能 |
| 5 | **P1-5 + P2-1/2-2** 补物理媒体键 globalShortcut + mediaSession `playbackState` + 修 play/pause 语义 | 中 | 播客核心体验（后台媒体键、系统卡片） |
| 6 | **P1-3** `powerMonitor('resume')` 重探网络/重试下载/唤醒续播 + `devicechange` | 中 | 睡眠恢复（笔记本合盖常态） |
| 7 | **P1-6 / P1-7** RSS 抓取加 `maxContentLength` + 解析移 Worker/分片 | 中 | 大档/海量刷新的内存与卡顿 |

> 第 1、2 项强烈建议**立刻**修：各只需补一个事件监听 / 一个 `.catch`，却能堵住"崩主进程"和"事故故障域每秒 rejection"两个真实高频隐患。

## 四、与材料对照：原生已满足 / 缺失 / 新引入

- **已做好、不用动**（agent 确认到位）：托盘完整（Tray + 右键菜单 + 状态跟随，Windows 路径完整）；沉浸页主题隔离干净（硬编码深色 + `::v-deep` 覆盖，零泄漏）；网络超时全设（RSS 20s / 下载 30s / NAS 探测 800ms / 发现 15-20s）；NAS 熔断 + 30s 重探 + 5min 心跳 + 看门狗切 CDN + 预取三闸节流；howler 切歌即 `Howler.unload()`（旧实例不累积）；DB `bulkGet/bulkPut` 单事务（无逐条写风暴）；`sanitizeHtml` 白名单 XSS 防护；`openDatabase` 事故加固 + `backups/` 轮转 10 份；feedUrl 校验 + OPML/RSS 嗅探；下载网络侧异常完整（仅写入侧缺，见 P1-1）。
- **缺失的平台能力**：Windows 物理媒体键 / SMTC `playbackState`、`powerMonitor`、退出 flush、最大化状态恢复、`devicechange` 设备热插拔。
- **新功能引入的新风险**：沉浸页双层滤镜合成开销（P1-8）、NAS 缓存无在途去重 + limit=500（P2-11）、coverColor/coverPalette 缓存无上限（P3-3）。
