## 一、可信度说明（对抗核实纠了什么）

审计阶段最有价值的，是对抗核实**主动证伪/降级**了一批被夸大的发现，避免按错误前提去优化：

### 被证伪（1 条）
- ❌ **Player Proxy 每秒触发一次 saveSelfToLocalStorage：把整个 _list/_currentTrack/_shuffledList 全量 JSON.stringify 写 localStorage(每秒一次, 与 B69-F5 心跳同域功耗)**（state-persistence / id=`player-proxy-per-second-localstorage-write`）
  - 证伪理由：该发现的承重论点——"Player.js:312 的 this._progress = cur 经过 Proxy → 每秒触发一次 saveSelfToLocalStorage() 全量序列化"——是对 JS Proxy + 闭包 this 行为的误读，与真实代码不符。 逐条核验： (1) store/index.js:55-64 的 Proxy set 拦截属实：对除 _howler 外的任意 prop 赋值都调 saveSelfToLocalStorage()+sendSelfToIpcMain()。 (2) Player.js:307-314 的 setInterval(…,1000) 每秒执行 this._progress = cur 也属实。 (3) 关键反驳:该 setInterval 回调的 this = **原始 Player 实例**,不是 Proxy。证据链:构造函数 line 121 调 _init() → line 282 调 _setIntervals() → line 307 注册 interval,整个过程发生在构造期;而 Proxy 是在 store/index.js:54-55 `new Player()` 返回后才用 new Proxy(player,…) 另建的一个**包装对象**,仅赋给 store.state.player。回调闭包捕获的 this 永远是原始实例(line 124 `window.yesplaymusic.player = this` 也指向原始实例)。JS Proxy 的 set trap 只在"通过 proxy 引用写属性"时触发;target 对自身内部的 this._progress = cur 写入**不经过 trap**。故每秒这次进度赋值根本不会触发 saveSelfToLocalStorage()。 (4) 每秒唯一经 Proxy 读 progress 的是组件里的只读 getter(Player.vue:911/937/1308、lyrics.vue:129 显示用),getter 不触发 set trap;v-model="player.progress"(Player.vue:21/498、lyrics.vue:132)只在用户拖动进度条时才走 setter。正常播放期(不拖条)没有任何"经 Proxy 的 progress 赋值"。 因此"每秒一次全量 _list/_currentTrack/_shuffledList JSON.stringify 落盘"这个核心断言不成立——它把内部直写误判成了经 Proxy 的写。承重的"每秒"频率被推翻。 附注(属实但非本条断言):saveSelfToLocalStorage (Player.js:1134-1142) 确实只排除 3 个 key(Player.js:48-52),会全量序列化 _list/_currentTrack 等,且确实会在"经 proxy 的真实状态变更"(如组件里 store.state.player.volume=…、换曲/换列表、appendTrackToPlayerList 等通过 proxy 引用赋值)时被调用。但这是**事件驱动**(换曲/改列表/调音量),不是每秒后台写,与 B69-F5 那种每秒后台写不同域,体量与频率均远低于发现所称。

### 关键降级 / 纠偏
- **"播放时每秒把整个 player 对象 JSON 序列化落盘"是个传说，已证伪**：每秒 `setInterval` 回调里的 `this` 是**裸 Player 实例**，不是 store 那层 Proxy（Proxy 在 `new Player()` 之后才包）。所以每秒写 `_progress` **不触发** set 陷阱、**不触发** `saveSelfToLocalStorage` 全量序列化——整对象序列化只在换曲/调音量等**离散事件**时发生。每秒真实成本只剩：`localStorage.setItem` 写一个数字 + 几个廉价 computed + **3 次 Dexie 写事务**（=真问题 B69-F5）。
- 多条 render 层发现（统计页 FLIP 重排、沉浸页滤镜、TrackListItem 重算、库卡片 hover blur）核实后**下调**：规模没那么大（如统计页行数=听过的**节目数**几十行，非机核 1000+ 单集；库卡片 hover 是逐卡 O(1) 非全网格）。
- 沉浸页滤镜机制纠偏：封面 scale 动画**不**喂 backdrop-filter，真正掉帧窗口是 **0.32s 进场过渡**。

### 1. getEpisodesByPodcast 用 sortBy 在内存排序整档单集(机核 1000+)

- **证据**：src/utils/podcast/db.js:182-188 getEpisodesByPodcast = db.episodes.where('podcastId').equals(podcastId).reverse().sortBy('pubTime')(注:文件在 podcast/ 子目录,原证据笼统写 db.js 但行号精确)。 Schema src/utils/db.js:47 episodes: '&id, podcastId, pubTime'(无 [podcastId+pubTime] 复合索引)。 package.json:38 dexie ^3.0.3。 rssParser.js:120 description 取更长文本,rssParser.js:26 capLen 上限 100KB;db.js:148-167(B-83)保留长 description。 详情侧: podcastDetail.vue:497 每次进详情都调;line 479-484 L1 缓存命中仍不 early-return,继续全量重读。 库页侧: podcastLibrary.vue:284-288 getLatestEpisodeTime 整档拉取取 eps[0];podcastLibrary.vue:532 对每档无条件执行;loadPodcasts 在 created()(line 436) 与 activated()(line 431,keep-alive)均触发 → 每次进"我的订阅"页都重跑。

### 3. parseRss 里 textNS/attrNS 对每个 item 反复 getElementsByTagName('*') 遍历全部后代，大 feed 接近 O(item × 后代数)

- **子系统**：`network-fetch`　**严重度**：high → **high**（低优·未核验）　**工作量**：M　**KNOWN**：审P1-7
- **证据**：rssParser.js textNS (:33-41) 与 attrNS (:43-51) 每次都 el.getElementsByTagName('*') 取该元素**所有后代**再线性扫。parseRss 主循环 (:94-127) 每个 item 调用 attrNS(item,'image','href')、textNS(item,'duration')、textNS(item,'encoded') 至少 3 次。当 item 含 <content:encoded> 富文本（小宇宙/机核 shownotes 是大段 HTML，编码后作为后代节点解析）时，单 item 的后代节点可能上千，每 item 被全量遍历 ~3 次。channel 级 attrNS/textNS (:84-86) 同理遍历整个文档后代。

## 六、推荐落地顺序

1. **数据层"整档重复读"集群**（第一梯队 + 第二梯队相关条）——最高性价比：改动 S–M、风险低、**每次进库页/进详情/进统计页都受益**。先做 podcasts 记录冗余 `latestPubTime` 字段（彻底干掉 `getLatestEpisodeTime` 整档读）+ `[podcastId+pubTime]` 复合索引（免内存 `sortBy`）+ 统计页去重扫/只读 `listenedSec` 不读 bits。
2. **B69-F5 心跳降频**——每秒 3 次 Dexie 写事务降为 5–10s 节流落盘（内存累加 + 暂停/切歌/退出强制 flush）。功耗/SSD 写放大。⚠️ 动统计须谨慎（审P2-10 刚事务化）。
3. **刷新条件请求（ETag/304）+ shownotes 预取节流**——网络/CPU 尖峰。
4. **F1 真窗口虚拟化**——专开一轮（曾因 B-74 滚动条/变高行被否，需先解滚动条稳定与变高行）；可先用 `Set`+computed 派生做"半步"降本。
5. **审P1-7 parseRss 移 Worker + 去重子树扫描**——专开一轮。

> ⚠️ 第 1 步含 **Dexie schema 版本号 bump**（加复合索引），对真实用户数据有风险——建议在 `feature/perf` 分支做、PodPlayerDev 实例验证 + 备份后再并入。
